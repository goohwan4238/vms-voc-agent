# PRD: 자재 입출고 이력 조회 화면

---

## 1. 개요

### 배경
현재 시스템에서 자재 입고/출고 처리는 가능하나, 이력을 통합적으로 조회할 수 있는 전용 화면이 없어 운영 실무자가 과거 이력 확인 시 불편을 겪고 있다.

### 목적
자재 입출고 이력을 다중 조건으로 검색·조회하고 엑셀로 내보낼 수 있는 이력 조회 화면을 신규 개발하여 업무 효율을 개선한다.

### 범위
- 자재관리 모듈 내 신규 조회 화면 1종
- 조회 API 엔드포인트 신규 개발
- 엑셀 다운로드 기능 (2차 배포)

## 2. 목표

| # | 목표 | 측정 기준 |
|---|------|----------|
| G1 | 이력 데이터를 3개 이상의 필터 조건으로 조회 가능 | 날짜/자재코드/유형 필터 모두 동작 |
| G2 | 조회 응답 시간 3초 이내 (최근 1년 기준) | API 응답 p95 ≤ 3s |
| G3 | 조회 결과를 엑셀 파일로 다운로드 가능 | `.xlsx` 정상 생성 및 다운로드 |
| G4 | 페이징 처리로 대용량 이력 안정적 표시 | 10만 건 이상 데이터에서도 정상 동작 |

## 3. 사용자 스토리

### US-01. 날짜 범위 조회
> **자재 담당자**로서, 특정 기간의 입출고 이력을 조회하고 싶다.  
> → 날짜 범위(시작일/종료일)를 입력하면 해당 기간의 이력 목록이 표시된다.

**수용 조건**
- 시작일/종료일 날짜 피커 제공
- 날짜 미입력 시 기본값: 최근 1개월
- 종료일이 시작일보다 이전이면 유효성 오류 표시

---

### US-02. 자재 코드 필터
> **자재 담당자**로서, 특정 자재의 이력만 골라 보고 싶다.  
> → 자재 코드 또는 자재명으로 검색하면 해당 자재의 이력만 필터링된다.

**수용 조건**
- 자재 코드 직접 입력 또는 자재명 부분 검색(like 검색) 지원
- 코드/명칭 중 하나만 입력해도 조회 가능

---

### US-03. 입출고 유형 필터
> **자재 담당자**로서, 입고 또는 출고 이력을 구분하여 조회하고 싶다.  
> → 유형 선택(전체/입고/출고)에 따라 결과가 필터링된다.

**수용 조건**
- 전체 / 입고 / 출고 3개 옵션 제공
- 기본값: 전체

---

### US-04. 이력 목록 확인
> **자재 담당자**로서, 조회 결과를 목록으로 한눈에 확인하고 싶다.  
> → 이력이 테이블 형태로 페이징 표시된다.

**수용 조건**
- 페이지당 기본 20건 표시, 50/100건 선택 가능
- 컬럼: 일시 / 자재코드 / 자재명 / 유형 / 수량 / 단위 / 담당자 / 비고
- 기본 정렬: 일시 내림차순

---

### US-05. 엑셀 다운로드 *(2차 배포)*
> **자재 담당자**로서, 조회 결과를 엑셀로 저장하여 보고서에 활용하고 싶다.  
> → 다운로드 버튼 클릭 시 현재 필터 조건의 전체 결과가 `.xlsx`로 다운로드된다.

**수용 조건**
- 현재 조회 조건 기준 전체 데이터 다운로드 (페이징 무관)
- 다운로드 최대 건수: 10,000건 (초과 시 경고 후 제한)
- 파일명 형식: `자재이력_YYYYMMDD_HHMMSS.xlsx`

## 4. 기능 명세

### 4.1 화면 레이아웃

```
┌─────────────────────────────────────────────────┐
│  자재 입출고 이력 조회                           │
├─────────────────────────────────────────────────┤
│  [날짜 From] ~ [날짜 To]  [자재코드/명] [유형▼] │
│                               [조회] [초기화]   │
├─────────────────────────────────────────────────┤
│  총 N건                         [엑셀 다운로드] │
│  ┌────┬──────┬──────┬──┬──┬──┬────┬────┐       │
│  │ 일시│코드  │자재명│유형│수량│단위│담당│비고│  │
│  └────┴──────┴──────┴──┴──┴──┴────┴────┘       │
│                     [< 1 2 3 ... >]             │
└─────────────────────────────────────────────────┘
```

### 4.2 API 명세

#### `GET /api/materials/transactions`
조회 API

| 파라미터 | 타입 | 필수 | 설명 |
|---------|------|------|------|
| `dateFrom` | `string` (YYYY-MM-DD) | N | 시작일 (기본: -1개월) |
| `dateTo` | `string` (YYYY-MM-DD) | N | 종료일 (기본: 오늘) |
| `materialCode` | `string` | N | 자재코드 (부분일치) |
| `materialName` | `string` | N | 자재명 (부분일치) |
| `transactionType` | `enum(ALL, IN, OUT)` | N | 입출고 유형 (기본: ALL) |
| `page` | `integer` | N | 페이지 번호 (기본: 1) |
| `pageSize` | `integer` | N | 페이지 크기 (기본: 20) |

**응답 예시**
```json
{
  "total": 1523,
  "page": 1,
  "pageSize": 20,
  "data": [
    {
      "transactionId": "TXN-20260201-001",
      "transactionAt": "2026-02-01T09:30:00",
      "materialCode": "MAT-001",
      "materialName": "볼트 M10",
      "type": "IN",
      "quantity": 500,
      "unit": "EA",
      "managerId": "U001",
      "managerName": "홍길동",
      "remark": "정기 발주 입고"
    }
  ]
}
```

#### `GET /api/materials/transactions/export`
엑셀 다운로드 API (2차 배포)

- 조회 파라미터 동일 (page/pageSize 제외)
- 응답: `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`

### 4.3 데이터 요구사항

**전제 조건 확인 필요 (개발 착수 전)**

```
[ ] material_transaction 테이블 존재 여부 확인
[ ] 이력 데이터 적재 시점 및 데이터 품질 확인
[ ] 테이블 미존재 시 → 로깅 구조 설계 선행 후 공수 재산정
```

**관련 테이블**

| 테이블 | 용도 | 비고 |
|--------|------|------|
| `material_transaction` | 입출고 이력 | 핵심 테이블 |
| `material_master` | 자재코드/명 조회 | 자재명 검색용 |
| `warehouse` | 창고 정보 | 선택적 표시 |

**필수 인덱스**
```sql
-- 날짜 범위 조회 성능 확보
CREATE INDEX idx_mat_txn_date ON material_transaction (transaction_at);

-- 자재코드 필터 성능 확보
CREATE INDEX idx_mat_txn_code ON material_transaction (material_code);

-- 복합 인덱스 (날짜 + 유형)
CREATE INDEX idx_mat_txn_date_type ON material_transaction (transaction_at, type);
```

## 5. 타 기능 영향도

### 5.1 영향 범위

| 구분 | 대상 | 영향 유형 | 대응 방안 |
|------|------|----------|----------|
| **모듈** | 자재관리 모듈 | 직접 연관 | 신규 메뉴 추가 |
| **연계 화면** | 자재 입고 처리 화면 | 데이터 원천 공유 | 읽기 전용, 변경 없음 |
| **연계 화면** | 자재 출고 처리 화면 | 데이터 원천 공유 | 읽기 전용, 변경 없음 |
| **연계 화면** | 재고 현황 조회 | 데이터 공유 | 영향 없음 |
| **DB** | `material_transaction` | 조회 쿼리 추가 | DML 없음, 인덱스 추가 |

### 5.2 기존 기능 변경 없음 확인
- 이력 조회 화면은 **읽기 전용(Read-Only)** 으로 기존 입출고 처리 로직에 영향 없음
- 인덱스 추가는 기존 쿼리 성능에 영향을 주지 않거나 개선함

## 6. 기술적 접근 방안

### 6.1 아키텍처

```
Frontend (조회 화면)
    │  HTTP GET
    ▼
Backend API (/api/materials/transactions)
    │  Query
    ▼
Database (material_transaction + index)
```

### 6.2 성능 전략

| 이슈 | 대응 방안 |
|------|----------|
| 대용량 날짜 범위 조회 | DB 인덱스 + 조회 기간 최대 1년 제한 |
| 페이징 성능 | `OFFSET` 대신 커서 기반 페이징 검토 (데이터 10만 건 이상 시) |
| 엑셀 다운로드 부하 | 최대 10,000건 제한 + 서버 스트리밍 방식 처리 |

### 6.3 엑셀 다운로드 구현 방안 *(2차)*

- **라이브러리**: `exceljs` (Node.js) 또는 `Apache POI` (Java) 등 기존 프로젝트 스택 따름
- **처리 방식**: 스트리밍 방식으로 메모리 최소화
- **10,000건 초과 시**: 경고 메시지 노출 후 사용자 확인 → 진행 또는 취소

### 6.4 권한 정책 확인 필요

```
[ ] 이력 열람 범위: 전체 이력 vs 본인 처리 이력만
[ ] 역할별 접근 권한 (관리자 / 담당자 / 조회 전용)
[ ] 엑셀 다운로드 권한 별도 부여 여부
```

## 7. 테스트 계획

### 7.1 단위 테스트

| 테스트 항목 | 조건 | 기대 결과 |
|------------|------|----------|
| 날짜 유효성 검증 | 종료일 < 시작일 | 400 오류 반환 |
| 날짜 유효성 검증 | 조회 범위 1년 초과 | 400 오류 반환 |
| 필터 없이 조회 | 파라미터 전부 기본값 | 최근 1개월 전체 결과 반환 |
| 자재코드 부분일치 | `materialCode=MAT` | `MAT-001`, `MAT-002` 등 포함 결과 |
| 유형 필터 | `transactionType=IN` | 입고 이력만 반환 |
| 페이징 | `page=2, pageSize=20` | 21~40번째 데이터 반환 |

### 7.2 통합 테스트

| 테스트 항목 | 시나리오 |
|------------|---------|
| 복합 필터 조회 | 날짜 + 자재코드 + 유형 동시 적용 |
| 결과 없음 처리 | 조건에 맞는 데이터 없을 때 빈 목록 + 안내 메시지 표시 |
| 엑셀 다운로드 | 조회 결과 1,000건 다운로드 → 파일 컬럼/데이터 정합성 확인 |
| 엑셀 제한 초과 | 10,001건 초과 시 경고 표시 확인 |

### 7.3 성능 테스트

| 테스트 항목 | 목표 |
|------------|------|
| 1년치 전체 조회 (10만 건 이상) | 응답 p95 ≤ 3초 |
| 동시 사용자 10명 조회 | 오류율 0% |
| 엑셀 10,000건 다운로드 | 30초 이내 완료 |

### 7.4 UAT (사용자 수용 테스트)

- 자재 담당자 1~2명 참여
- 실제 운영 데이터 기반 조회 시나리오 검증
- 화면 UX 피드백 반영

## 8. 일정

### 전제: 이력 데이터 적재 확인 완료 기준

| 단계 | 작업 내용 | 기간 | 담당 |
|------|----------|------|------|
| **0. 사전 확인** | DB 이력 테이블 존재 여부 및 데이터 품질 확인 | 0.5일 | 개발 + DBA |
| **1. 설계** | API 설계, DB 인덱스 설계, 화면 와이어프레임 | 0.5일 | 개발 |
| **2. 백엔드 개발** | 조회 API 개발 + 인덱스 적용 | 1.5일 | 백엔드 |
| **3. 프론트엔드 개발** | 조회 화면 + 필터 UI + 그리드 구현 | 1.5일 | 프론트엔드 |
| **4. 테스트 및 수정** | 단위/통합/성능 테스트 + 버그 수정 | 1일 | 개발 전체 |
| **5. 배포 (1차)** | 조회 기능 운영 배포 | 0.5일 | DevOps |
| **6. 엑셀 다운로드 (2차)** | 엑셀 기능 개발 → 테스트 → 배포 | 1~2일 | 개발 전체 |

> **총 예상 공수**: 1차 배포 기준 **4~5일** / 엑셀 포함 2차 기준 **5~7일**  
> ⚠️ 이력 테이블 미존재 시 DB 설계 및 데이터 마이그레이션 추가로 **+3~5일** 재산정 필요
