# 결재 문서 목록 검색 기능 추가 — PRD

---

## 1. 개요

결재 문서 수 증가에 따라 사용자가 원하는 문서를 신속하게 탐색하기 어려운 문제가 발생하고 있다. 본 PRD는 결재 문서 목록 화면에 키워드 검색, 날짜 범위 필터, 결재 상태 필터를 추가하여 사용자의 문서 탐색 효율을 높이는 것을 목적으로 한다.

| 항목 | 내용 |
|------|------|
| **기능명** | 결재 문서 목록 검색 및 필터 |
| **요청 출처** | VOC — 결재 문서 목록에서 검색 기능 추가 요청 |
| **우선순위** | Medium |
| **예상 공수** | Medium |
| **개발 권고** | 개발 ✅ |

---

## 2. 목표

### 비즈니스 목표
- 결재 문서 탐색 시간 단축으로 사용자 생산성 향상
- 문서 수 증가 추세에 선제적으로 대응하여 시스템 사용성 유지

### 성공 지표
- 결재 문서 목록 페이지에서 검색 기능 사용률 측정
- 문서 탐색 관련 VOC 재발생 여부 모니터링
- 검색 쿼리 평균 응답 시간 2초 이내 유지

---

## 3. 사용자 스토리

### US-01. 키워드로 문서 검색
> **As** 결재 시스템 사용자,  
> **I want** 문서 제목 또는 작성자명으로 키워드 검색을,  
> **So that** 많은 문서 중에서 원하는 문서를 빠르게 찾을 수 있다.

**수용 기준**
- 검색창에 키워드 입력 후 Enter 또는 검색 버튼 클릭 시 결과 즉시 반영
- 제목, 작성자명 중 하나라도 키워드를 포함하는 문서가 결과에 표시됨
- 검색어가 비어 있으면 전체 목록 표시

---

### US-02. 날짜 범위로 문서 조회
> **As** 결재 시스템 사용자,  
> **I want** 작성일 또는 결재일 기준으로 기간을 지정하여 문서를 조회하고 싶다,  
> **So that** 특정 시점의 문서만 선별해서 확인할 수 있다.

**수용 기준**
- 날짜 유형(작성일 / 결재일) 선택 옵션 제공
- 시작일과 종료일을 각각 선택 가능하며, 둘 중 하나만 입력해도 동작
- 잘못된 날짜 범위(종료일 < 시작일) 입력 시 에러 메시지 표시

---

### US-03. 결재 상태로 필터링
> **As** 결재 시스템 사용자,  
> **I want** 결재 상태(임시저장 / 결재중 / 승인 / 반려)를 선택하여 목록을 필터링하고 싶다,  
> **So that** 처리가 필요한 문서나 특정 상태의 문서만 모아 볼 수 있다.

**수용 기준**
- 상태 필터는 다중 선택 가능
- 선택된 상태에 해당하는 문서만 목록에 표시
- 필터 초기화 버튼 클릭 시 전체 상태 표시로 복귀

---

### US-04. 검색 조건 복합 적용
> **As** 결재 시스템 사용자,  
> **I want** 키워드, 날짜 범위, 상태 필터를 동시에 조합하여 검색하고 싶다,  
> **So that** 정밀한 조건으로 문서를 빠르게 찾을 수 있다.

**수용 기준**
- 모든 검색 조건은 AND 조건으로 결합
- 각 조건 변경 시 목록이 즉시 재조회됨
- 적용된 필터 조건을 UI 상단에 태그 형태로 표시하고 개별 제거 가능

## 4. 기능 명세

### 4.1 검색 UI 구성

| 컴포넌트 | 타입 | 설명 |
|----------|------|------|
| 키워드 검색창 | Text Input | 제목·작성자 통합 검색. placeholder: "제목 또는 작성자 검색" |
| 날짜 유형 선택 | Select Box | 작성일 / 결재일 |
| 날짜 범위 | Date Picker × 2 | 시작일 ~ 종료일 |
| 상태 필터 | Multi-select Checkbox | 임시저장, 결재중, 승인, 반려 |
| 검색 버튼 | Button | 조건 확정 및 조회 실행 |
| 초기화 버튼 | Button | 모든 조건 초기화 |
| 적용 필터 태그 | Tag List | 현재 적용된 조건 표시 및 개별 제거 |

### 4.2 검색 조건 파라미터 정의

| 파라미터 | 타입 | 필수 여부 | 설명 |
|----------|------|-----------|------|
| `keyword` | string | 선택 | 제목·작성자명 키워드 (부분 일치) |
| `date_type` | enum | 선택 | `created_at` \| `approved_at` |
| `date_from` | date (YYYY-MM-DD) | 선택 | 조회 시작일 |
| `date_to` | date (YYYY-MM-DD) | 선택 | 조회 종료일 |
| `status` | enum[] | 선택 | `draft` \| `pending` \| `approved` \| `rejected` |
| `page` | integer | 필수 | 페이지 번호 (default: 1) |
| `page_size` | integer | 필수 | 페이지당 건수 (default: 20) |

### 4.3 검색 결과 표시

- 검색 결과 건수를 목록 상단에 표시 (예: "총 38건")
- 검색 결과가 0건일 때 "검색 결과가 없습니다" 안내 문구 표시
- 기존 목록 정렬(최신순 등) 기능과 검색 필터는 독립적으로 동작
- 페이지 이동 시 현재 검색 조건 유지

### 4.4 권한별 검색 범위

| 역할 | 검색 범위 |
|------|-----------|
| 일반 사용자 | 본인이 작성하거나 본인이 결재자로 포함된 문서 |
| 관리자 | 전체 문서 |

> 권한 범위는 기존 접근 제어 정책을 따르며, 검색 조건 적용 전 서버에서 권한 필터를 우선 적용한다.

## 5. 타 기능 영향도

### 5.1 영향 모듈

| 모듈 | 변경 유형 | 상세 |
|------|-----------|------|
| 결재 문서 목록 | UI 변경 + API 연동 수정 | 검색 UI 추가, API 파라미터 연동 |
| 결재 상태 관리 | 참조 | 상태 코드값 확인 및 정규화 여부 검토 |
| 사용자 관리 | DB 조인 | 작성자 검색 시 `users` 테이블 조인 |

### 5.2 DB 영향도

| 테이블 | 변경 내용 |
|--------|-----------|
| `approval_documents` | `title`, `created_at`, `approved_at` 컬럼 인덱스 추가 검토 |
| `users` | `name` 컬럼 인덱스 추가 검토 |
| `approval_status` | 상태 코드 정규화 여부 확인 |

### 5.3 API 변경

```
GET /approvals
  기존: page, page_size, sort
  추가: keyword, date_type, date_from, date_to, status[]
```

기존 파라미터 없이 호출하는 클라이언트는 동작에 변화 없음 (하위 호환 유지).

## 6. 기술적 접근 방안

### 6.1 텍스트 검색 전략

**1단계 (기본)**: DB LIKE 검색
- `title LIKE '%keyword%'` 또는 `users.name LIKE '%keyword%'`
- 문서 수 적을 때 충분하나, 데이터 증가 시 풀스캔 리스크 존재

**2단계 (성능 임계 도달 시)**: Full-Text Search 적용 또는 검색 전용 인덱스 구성
- PostgreSQL 사용 시: `tsvector` + `GIN Index`
- MySQL 사용 시: `FULLTEXT INDEX`
- 또는 Elasticsearch 등 외부 검색 엔진 도입 검토

> **권고**: 1단계로 출시 후 쿼리 성능 모니터링 결과에 따라 2단계 적용 여부 결정.

### 6.2 날짜 범위 필터

```sql
-- date_type = 'created_at' 예시
WHERE created_at BETWEEN :date_from AND :date_to
```

- `created_at`, `approved_at` 컬럼에 B-Tree 인덱스 적용

### 6.3 상태 필터

```sql
WHERE status IN (:status_list)
```

- 상태 코드 정규화 완료 전제. 정규화 미완료 시 별도 매핑 레이어 추가 필요

### 6.4 권한 필터 우선 적용

```sql
-- 일반 사용자
WHERE (created_by = :user_id OR approver_id = :user_id)
  AND /* 검색 조건 */

-- 관리자
WHERE /* 검색 조건만 */
```

### 6.5 페이지네이션

- `LIMIT / OFFSET` 기반 유지 (기존 방식 계승)
- 검색 결과 건수는 별도 COUNT 쿼리 또는 캐시 처리

## 7. 테스트 계획

### 7.1 단위 테스트

| 대상 | 케이스 |
|------|--------|
| 키워드 검색 | 제목 일치, 작성자 일치, 둘 다 일치, 결과 없음 |
| 날짜 필터 | 시작일만, 종료일만, 범위 내, 범위 외, 잘못된 범위 |
| 상태 필터 | 단일 선택, 다중 선택, 전체 선택, 선택 없음 |
| 복합 조건 | 키워드 + 날짜 + 상태 조합 |

### 7.2 권한 테스트

| 시나리오 | 기댓값 |
|----------|--------|
| 일반 사용자가 타인 문서 키워드로 검색 | 해당 문서 미노출 |
| 관리자가 동일 키워드로 검색 | 전체 문서 노출 |

### 7.3 성능 테스트

| 항목 | 기준 |
|------|------|
| 키워드 검색 응답 시간 | 데이터 10만 건 기준 2초 이내 |
| 날짜 + 상태 복합 필터 응답 시간 | 2초 이내 |

### 7.4 UX 테스트

- 기존 다른 목록 화면의 검색 UI 패턴과 일관성 확인
- 모바일 뷰포트에서 필터 패널 가시성 확인
- 빈 결과 상태 안내 문구 확인
