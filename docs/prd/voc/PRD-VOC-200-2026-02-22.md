## 1. 개요

### 1.1 배경

현재 프로젝트 일정은 텍스트 기반 목록 형태로 표시되어, 태스크 간 시간적 관계와
전체 프로젝트 진행 현황을 직관적으로 파악하기 어렵다.
사용자 피드백(VOC)을 통해 시각적 일정 관리 도구에 대한 수요가 확인되었다.

### 1.2 문제 정의

- 텍스트 목록만으로는 태스크 간 의존 관계 및 타임라인 파악이 어려움
- 프로젝트 전체 진행률을 한눈에 확인할 수 없음
- 담당자별 업무 부하(workload) 파악이 불가능함
- 마일스톤 도달 시점을 시각적으로 인지하기 어려움

### 1.3 범위

| 구분 | 내용 |
|------|------|
| 포함 (MVP) | 타임라인 기반 간트차트, 태스크별 진행률 표시 |
| 포함 (MVP) | 마일스톤 마커 표시 |
| 2차 릴리즈 | 담당자 필터, 마일스톤 상세 알림 |
| 제외 | 간트차트 내 드래그앤드롭 편집, 모바일 최적화 뷰 |

## 2. 목표

### 2.1 비즈니스 목표

- 프로젝트 일정 시각화를 통해 팀 내 커뮤니케이션 비용 절감
- 프로젝트 관리 도구로서의 핵심 가치 강화로 사용자 리텐션 개선

### 2.2 제품 목표

- 사용자가 프로젝트 전체 타임라인을 5초 이내에 파악할 수 있도록 한다
- 각 태스크의 진행률을 차트 내에서 직접 확인할 수 있도록 한다
- 마일스톤을 시각적 마커로 구분하여 중요 이벤트를 즉시 인지할 수 있도록 한다

### 2.3 성공 지표 (KPI)

| 지표 | 목표치 |
|------|--------|
| 간트차트 뷰 월간 활성 사용자(MAU) | 전체 사용자의 60% 이상 |
| 간트차트 평균 페이지 로드 시간 | 2초 이하 (태스크 100건 기준) |
| 기능 출시 후 일정 관련 VOC 감소율 | 30% 이상 |

## 4. 기능 명세

### 4.1 간트차트 시각화 (MVP)

#### 4.1.1 타임라인 바(Bar) 렌더링

- **조건**: `tasks.start_date` 및 `tasks.end_date`가 존재하는 태스크만 표시
- **날짜 축**: 일(Day) / 주(Week) / 월(Month) 단위 전환 가능
- **줌(Zoom)**: 날짜 범위 확대·축소 지원
- **태스크 행 구성**:
  - 행 레이블: 태스크명, 담당자 아바타
  - 바(Bar): `start_date` ~ `end_date` 구간에 맞게 렌더링
  - 바 색상: 태스크 상태(예정/진행/완료/지연)에 따라 구분

#### 4.1.2 진행률 표시

- 바(Bar) 내부에 완료율(%) 오버레이 표시
- 완료율 데이터 소스: `tasks.progress` 컬럼 (0~100 정수)
- 완료율에 따라 바 내부 채움(fill) 비율 시각적으로 표현

#### 4.1.3 마일스톤 마커 (MVP)

- 마일스톤은 바(Bar)가 아닌 다이아몬드(◆) 마커로 타임라인 위에 표시
- 마커 클릭 시 마일스톤명, 날짜, 설명 툴팁 표시
- 데이터 소스: `milestones` 테이블

### 4.2 담당자 필터 (2차 릴리즈)

- 담당자 목록 드롭다운으로 특정 사용자의 태스크만 필터링
- 다중 선택 지원

### 4.3 마일스톤 알림 (2차 릴리즈)

- 마일스톤 D-3, D-0 시점에 담당자에게 인앱 알림 발송

## 6. 기술적 접근 방안

### 6.1 간트차트 라이브러리 선택

| 옵션 | 라이선스 | 장점 | 단점 |
|------|---------|------|------|
| **Frappe Gantt** | MIT | 경량, 커스터마이징 용이 | 기능 제한적 |
| **AG Grid Gantt** | Community/Enterprise | 기능 풍부 | Enterprise 유료 |
| **DHTMLX Gantt** | GPL/Commercial | 완성도 높음 | 상용 비용 발생 |
| **자체 구현 (SVG/Canvas)** | - | 완전한 제어권 | 개발 공수 大 |

> **권고**: MVP에서는 **Frappe Gantt (MIT)** 도입. 기능 한계 도달 시 자체 구현 전환.

### 6.2 성능 최적화

- **가상화(Virtualization)**: 태스크 200건 이상 시 가상 스크롤 적용 (react-window 등)
- **페이지네이션**: API에서 태스크를 페이지 단위로 로드, 무한 스크롤 지원
- **캐싱**: 간트차트 데이터 클라이언트 사이드 캐싱 (SWR / React Query)

### 6.3 데이터베이스 마이그레이션

```sql
-- tasks 테이블 컬럼 추가
ALTER TABLE tasks
  ADD COLUMN start_date DATE,
  ADD COLUMN end_date DATE,
  ADD COLUMN progress INT NOT NULL DEFAULT 0
    CHECK (progress >= 0 AND progress <= 100);

-- milestones 테이블 신규 생성
CREATE TABLE milestones (
  id          SERIAL PRIMARY KEY,
  project_id  INT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  name        VARCHAR(255) NOT NULL,
  due_date    DATE NOT NULL,
  description TEXT,
  created_at  TIMESTAMP DEFAULT NOW(),
  updated_at  TIMESTAMP DEFAULT NOW()
);
```

### 6.4 반응형 전략

- **데스크톱** (≥1024px): 풀 간트차트 뷰
- **태블릿** (768~1023px): 날짜 축 축소, 스크롤 지원
- **모바일** (<768px): 간트차트 숨김, 목록 뷰(List View) 자동 전환 (2차에서 전용 뷰 설계)

## 7. 테스트 계획

### 7.1 단위 테스트

| 대상 | 테스트 항목 |
|------|-----------|
| `GET /projects/:id/tasks` | `start_date`, `end_date`, `progress` 필드 정상 반환 |
| `GET /projects/:id/milestones` | 마일스톤 목록 정상 반환, 프로젝트 미존재 시 404 |
| `PATCH /tasks/:id/progress` | 0~100 범위 검증, 범위 초과 시 400 반환 |
| DB 마이그레이션 | 기존 태스크 데이터 정합성 유지 확인 |

### 7.2 통합 테스트

| 시나리오 | 기대 결과 |
|---------|---------|
| 태스크 `start_date` / `end_date` 설정 후 간트차트 조회 | 해당 기간에 바(Bar) 렌더링 |
| 마일스톤 등록 후 간트차트 조회 | 해당 날짜에 다이아몬드 마커 표시 |
| 태스크 진행률 50% 설정 | 바(Bar) 내부 50% 채움 표시 |
| `start_date` 없는 태스크 | 간트차트에서 해당 행 미표시 (또는 별도 섹션 표시) |

### 7.3 성능 테스트

| 시나리오 | 목표 |
|---------|------|
| 태스크 100건 간트차트 초기 로드 | 2초 이내 |
| 태스크 500건 스크롤 | 프레임 드롭 없이 60fps 유지 |

### 7.4 브라우저 호환성
